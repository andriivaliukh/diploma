## Слайд 3 — Актуальність тематики · 34 с

- WireGuard автентифікує **статичні ключі, а не людей** (Cryptokey Routing).
- IPsec / OpenVPN → інтегрують EAP / RADIUS / LDAP. WireGuard → лише ключ.
- Ключова теза: **«ключ ≠ людина»**.
- Вербальний міст у сл. 4 (замість ✗-списку): «немає 2FA, немає динамічного відкликання, немає журналу входів, немає перевірки особи».
- Ризик: скомпрометований ноутбук → ключ лишається валідним; середовища під аудитом (SOC 2 CC6.1 / ISO 27001 A.9.4.2).
- ⚠ guard: це **свідоме рішення** WireGuard — мінімальний примітив рівня даних (~4000 LOC).
- ⚠ guard: **«формально верифікований» = протокол Noise / криптомодель (Donenfeld), НЕ вся реалізація ядра.** Мати відповідь напоготові.

## Слайд 5 — Ідея (запропонований підхід) · 40 с · ★ символ · **Реп. 1 (повна теза)**

- **«Сесія як вузол»** — peer існує на `wg0` лише за чинної 2FA-сесії.
- Додати: `wg set wg0 peer …` (вузол з'являється на `wg0` після 2FA). Зняття — проговорити усно: при завершенні/відкликанні сесії `… peer … remove` (на слайді — дрібна виноска).
- Якір: **«Без зміни протоколу. Без PSK.»** Збоку: «два subprocess-виклики».
- Контраст із DefGuard: у них PSK-токен + висока складність; у нас «вузол або є, або ні», низька складність.
- ⚠ guard (критичний — A1/Q2): **«Без PSK» = не використовуємо PSK як ТОКЕН СЕСІЇ (на відміну від DefGuard). НЕ прибираємо криптошар** — опційний статичний PSK WireGuard лишається доступним клієнтові.
- ⚠ guard: session-gated — новий тунель неможливий без запису peer на сервері.
- ⚠ guard (відкликання): синхронне через API — миттєве (зняття peer); фонове очищення прострочених — **до 60 с**. НЕ казати «миттєве відкликання» без уточнення.

## Слайд 6 — Як працює (процес автентифікації) · 75 с · ★ щільний · **Реп. 2 (механіка)**

- Повний двоетапний вхід + тунель. 3 доріжки: Клієнт ↔ API ↔ WireGuard. Реєстрацію — лише згадати.
- Проговорити 5 обмінів:
  1. → `login (пароль)` (Argon2id).
  2. ← проміжний токен · **`5 хв`**.
  3. → `TOTP` (вікно ±30 с).
  4. ← повний токен · **`24 год`**.
  5. → `сесія` → `wg set wg0 peer …` → ⇒ тунель (UDP :51820). «тут — вузол-сесія» (зв'язок зі сл. 5).
- Реп. 2: «механічно — це **два виклики subprocess**».
- ⚠ guard (крипто, для Q&A, зняте зі слайда): TOTP = RFC 6238, вікно ±1 крок (±30 с); JWT = **HS256**, один секрет, розрізнення за claim `scope`; **Fernet** шифрує TOTP-секрет **у спокої** (ключ з env через HKDF), у flow логіну НЕ бере участі.
- ⚠ guard: проміжний токен дійсний **виключно** для `/auth/totp/verify`; обхід 2FA на рівні API виключено (без TOTP → 403).
- ⚠ guard (A5/Q4): **stateless JWT — blacklist немає свідомо.** Відкликання VPN = зняття peer з `wg0` (трафік обрізаний навіть із валідним JWT); глобальний blacklist токена — напрям доопрацювання.
- ⚠ guard (A6/Q5 — НАЙНЕБЕЗПЕЧНІШЕ): **TOTP replay у вікні ±30 с** — серверне single-use споживання коду НЕ задокументоване. НЕ казати «реплей неможливий». Чесно: короткий TTL проміжного токена + HTTPS; маркування коду — напрям доопрацювання.
