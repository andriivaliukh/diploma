## Палітра та кодування (успадковують beamer-dev і diagram-dev — НЕ відступати)

**Сувора 3-фарбова схема (чорний + один акцент + сірий), безпечна для проєктора і дальтонізму:**
- Тло: `#FFFFFF` (або ледь теплий `#FCFCFA`). Без темних тем.
- Основна туш (текст, лінії діаграм, «наявний» вузол): майже-чорний `#1A1A1A`.
- **Єдиний акцент** (callout-стрілки, стовпець «Наша система», якірні підписи, замок 2FA):
  академічний синій **`#1F3A5F`**.
- Вторинний / «знятий» / негатив: середній сірий **`#7A7A7A`**.
- **✓ / ✗ — кодувати ФОРМОЮ гліфа в першу чергу:** жирний ✓ vs жирний ✗ (читаються у ч/б).
  Колір лише підсилення: ✓ = синій, ✗ = сірий. **Червоно-зелене НЕ використовувати.**
- **Додати vs зняти вузол (сл. 4/5):** додати = **суцільна синя стрілка + залитий вузол**;
  зняти = **пунктирна сіра стрілка + порожній сірий вузол**. Несе сигнал **стиль лінії, не колір**.
- Named TikZ-стилі (визначити один раз у преамбулі, перевикористати на сл. 4/5/6/7):
  `wgbus`, `wgpeer` (наявний), `wggone` (знятий), `callout`, `lock`. Шина `wg0` — однакова товщина скрізь.
- Кегль: тіло ≥40 pt на проєкції; пройти thumbnail-тест. Engine — XeLaTeX + fontspec (кирилична гарнітура),
  моно-шрифт із великою x-висотою. Тема `default`, зняти navigation/footline/headline/frametitle/logo, 4:3.

## Winston Star (ядро, що чіпляє)

- **SLOGAN / обіцянка (вербально на титулі, КОРОТКА форма ~12–15 слів):**
  > **«За п'ять хвилин покажу, як зробити обов'язкову 2FA поверх стандартного WireGuard —
  > без зміни протоколу і без PSK.»**
  Повна форма теми («сесія існує як вузол інтерфейсу; PSK не використовується як токен сесії»)
  **не вимовляється на титулі** — розгортається на сл. 4. Анкер-фраза для повтору: «Сесія як вузол».

- **SYMBOL (підпис-мотив):** горизонтальна шина **`wg0`**, на якій **вузол-сесія (peer) з'являється
  лише після проходження «замка» 2FA** (показано ОДИН суцільний вузол; зняття — дрібна виноска, не другий стан).
  Мотив «замок → вузол на `wg0`» повторюється: сл. 4 (повна форма), сл. 5 (крок `wg set wg0 peer …`),
  сл. 6 (площина даних), мінімізовано — сл. 7 (кутовий підпис).

- **SURPRISE (лідуючий результат сл. 6):** площина даних **ідентична** чистому WireGuard —
  паритет у межах **1 %** (tcp_t 890→881 Мбіт/с) і медіани рівні (tcp_p 917=917); ціна 2FA —
  **одноразові 1571 мс** онбордингу проти 92 мс, і тільки у площині керування.

- **SALIENT IDEA:** немодифікований протокол; контроль доступу винесено в керуючу площину
  через додавання/видалення вузла `wg0`.

- **STORY:** проблема (ключ ≠ людина) → ніхто з розглянутих не закриває всі 4 вимоги →
  ідея (сесія як вузол) → як працює → що це коштує → що зроблено.

## Слайд 2 — «Проблема»
- **Одна ідея:** WireGuard автентифікує **статичні ключі, а не користувачів**.
- **Діаграма (НОВА, проста — дві доріжки контрасту):**
  - Верхня — **IPsec / OpenVPN**: `[Користувач]` → `[EAP / RADIUS / LDAP]` → `[Тунель]`.
    Підпис: «інтегрують автентифікацію користувача».
  - Нижня — **WireGuard**: `[Пакет]` → `[Перевірка статичного публічного ключа]` (Cryptokey Routing)
    → `[Тунель]`. Запечена **callout-стрілка** (акцент-синій) до блоку перевірки: **«ключ ≠ людина»**.
  - **R2: 4-пунктовий ✗-список ПРИБРАНО зі слайда** (дублював рядки матриці сл. 3) — проговорити вербально як міст у сл. 3.
- **Текст на слайді (мінімум слів):**
  - «IPsec / OpenVPN → EAP / RADIUS / LDAP»
  - «WireGuard → лише ключ»  •  **«ключ ≠ людина»**
- **Час:** 40 с.
- **Цикл:** готує ґрунт (ще не теза).
- **Нотатки доповідача (гарди):**
  - Свідоме рішення WireGuard (мінімальний примітив площини даних, ~4000 LOC).
  - **«Формально верифікований» = протокол Noise / криптомодель (Donenfeld), НЕ вся реалізація ядра** — мати відповідь напоготові.
  - Вербальний міст у сл. 3 (замість ✗-списку): «немає 2FA, немає динамічного відкликання, немає журналу входів, немає перевірки особи».
  - Ризик: скомпрометований ноутбук — ключ лишається валідним; середовища під аудитом (SOC 2 CC6.1 / ISO 27001 A.9.4.2).

## Слайд 3 — «Конкуренти» (паркан)
- **Одна ідея:** жодне з розглянутих self-hosted рішень не закриває **всі 4 вимоги** одночасно.
- **Діаграма (R2: компактна матриця ✓/✗ — 4 РЯДКИ × 4 СТОВПЦІ на слайді):**
  - Стовпці (4): **DefGuard** (найближчий, тонка рамка) · **NetBird** (представник OIDC-залежних) ·
    **WAG** · **Наша система** (виділена заливкою акцент-синім).
  - Рядки (1–2 слова): **«WG-клієнт»** · **«Без IdP»** · **«Власний TOTP»** · **«Ключі»**
    (повні вимоги проговорити вербально; полярність **✓ = добре**).
  - Заповнення (канон §7):
    - DefGuard: WG-клієнт ✗ · Без IdP ✓ · TOTP ✓ · Ключі ✓
    - NetBird: WG-клієнт ✗ · **Без IdP ✗** · **Власний TOTP ✗** (MFA делеговано зовнішньому OIDC-IdP — не власний механізм) · Ключі ✓
    - WAG: WG-клієнт ✓ · Без IdP ✓ · TOTP ✓ · **Ключі ✗**
    - **Наша система: ✓ ✓ ✓ ✓** (єдиний повний стовпець)
  - Запечена **callout-стрілка** до стовпця «Наша система»: «єдиний повний стовпець ✓».
  - **✓/✗ — гліф-форма (жирний ✓ vs ✗), колір лише підсилення.** Повна матриця 5×4 (з Firezone та
    Tailscale/Headscale) → **резервний слайд** (Q&A).
- **Текст на слайді:** лише шапка 4×4 + позначки ✓/✗. Жодних речень.
- **Час:** 42 с.
- **Цикл:** будує паркан — «на відміну від DefGuard: **без PSK-токена** і **без власного клієнта**».
- **Нотатки доповідача (гарди):**
  - Критерії відбору: open-source, self-hosted, активні у 2025, явна WG-інтеграція.
  - **Скоупінг новизни — ОБОВ'ЯЗКОВО вербально:** «**серед розглянутих self-hosted рішень за цими
    чотирма критеріями**» — НІКОЛИ не казати голе «ми єдині/перші».
  - Firezone і NetBird мають ту саму OIDC-залежність від зовнішнього IdP — згадати вербально.
  - У рецензованій літературі систематичного підходу 2FA+WireGuard **немає — лише практичні реалізації**

## Слайд 4 — «Ідея» (наукова новизна)  ★ символ
- **Одна ідея:** **«Сесія як вузол»** — peer існує на `wg0` лише за чинної 2FA-сесії.
- **Діаграма (НОВА — це SYMBOL деки):**
  - Шина **`wg0`** з вузлом `[Сервер 10.10.0.1]` ліворуч.
  - По центру — **«замок 2FA»** (ворота, акцент-синій). Зліва від замка — `[Клієнт + публічний ключ]`,
    що сам по собі **не дає тунелю** (підпис: «ключа недостатньо»).
  - Праворуч від замка, НА шині `wg0`, **ОДИН суцільний залитий вузол `[Сесія-peer 10.10.0.x]`** —
    єдиний показаний стан. Зверху одна дія-стрілка (форма команди, БЕЗ аргументів):
    **`wg set wg0 peer …`** (суцільна синя стрілка, «сесія створена»).
  - **R3 (спрощення): другий, пунктирний «знятий» вузол ПРИБРАНО** (плутав — два стани одночасно).
    Зняття показано **дрібною сірою виноскою-футноутом** біля вузла: «× при завершенні сесії: `… peer … remove`» —
    НЕ другий вузол, НЕ конкуруюча стрілка; читається як примітка.
  - Потік однозначно зліва→направо, один шлях: `Клієнт+ключ` →(недостатньо)→ `2FA` → `peer на wg0`.
  - Якір-підпис великим: **«Без зміни протоколу. Без PSK.»**
- **Текст на слайді:**
  - **«Сесія як вузол»**
  - `wg set wg0 peer …` (додати)  ·  дрібна виноска: «× при завершенні: `… peer … remove`»
  - **Без зміни протоколу • Без PSK**
- **Час:** 40 с.
- **Цикл:** **Реп. 1 (повна теза).**
- **Нотатки доповідача (гарди):**
  - **«Без PSK» = не використовуємо PSK як ТОКЕН СЕСІЇ (на відміну від DefGuard), НЕ прибираємо
    криптографічний шар.** Опційний статичний PSK WireGuard лишається доступним клієнтові. *(критичний гард — A1/Q2)*
  - session-gated: новий тунель неможливий без запису peer на сервері.
  - **Відкликання: синхронне через API — миттєве (зняття peer); фонове очищення прострочених — до 60 с.
    НЕ казати «миттєве відкликання» без уточнення.**
  - Контраст із DefGuard (PSK-токен, висока складність) — у нас «вузол або є, або ні», низька складність.
  - **Після цього слайда — вербальна пунктуація** (див. вище).

## Слайд 5 — «Як працює» ★ ЩІЛЬНИЙ (hapax legomenon)
- **Одна ідея:** повний потік двоетапного входу і встановлення тунелю.
- **Діаграма (адаптує `fig:auth_sequence`, chapter2.tex р.223–305; R2: розущільнено до 5 обмінів):**
  3 доріжки **Клієнт ↔ API Server ↔ WireGuard**. Реєстрацію — лише згадати вербально.
  **Мітки стрілок ≤3 токени (дієслово, не маршрут); на слайді — тільки 2 TTL-анотації:**
  1. → **`login (пароль)`** → сервер: Argon2id (у нотатці).
  2. ← **проміжний токен** · анотація **`5 хв`**.
  3. → **`TOTP`** (±30 с — у нотатці).
  4. ← **повний токен** · анотація **`24 год`**.
  5. → **`сесія`** → API → **`wg set wg0 peer …`** (стрілка до WireGuard) → ⇒ **тунель (UDP :51820)**
     (подвійна лінія Клієнт↔WireGuard). Кроки 5–6 склеєно в один раунд-тріп.
  - **R2: бічні плашки (RFC 6238 / Fernet / HS256) ЗНЯТО зі слайда** → лише у нотатках / Q&A.
  - **R3: callout-стрілку «тут — вузол-сесія» ПРИБРАНО** (крок `wg set wg0 peer …` лишається на лінії
    WireGuard; зв'язок «вузол-сесія» доповідач озвучує вербально — без запеченої виноски на слайді).
- **Текст на слайді:** мітки-дієслова + дві TTL-анотації (`5 хв` / `24 год`). Без маршрутів, без JSON, без речень.
- **Час:** 75 с.
- **Цикл:** **Реп. 2** — «механічно: два виклики subprocess».
- **Нотатки доповідача (гарди — тут живуть зняті зі слайда деталі):**
  - **Криптодеталі (для Q&A):** TOTP = RFC 6238, вікно ±1 крок (±30 с); JWT = **HS256**, один секрет,
    розрізнення за claim `scope` у middleware; **Fernet** шифрує секрет TOTP **у спокої** (ключ з env через HKDF),
    НЕ бере участі у flow логіну.
  - Проміжний токен дійсний **виключно** для `/auth/totp/verify`; обхід 2FA на рівні API виключено (без підтвердження → 403).
  - **Stateless JWT — немає blacklist (теза свідомо):** відкликання VPN = зняття peer з `wg0` (трафік
    обрізаний навіть із валідним JWT); глобальний blacklist повного токена — напрям доопрацювання. *(A5/Q4)*
  - **TOTP replay у вікні ±30 с:** серверне single-use споживання коду НЕ задокументоване — НЕ стверджувати
    «реплей неможливий»; чесно: короткий TTL проміжного токена + HTTPS, маркування коду — напрям доопрацювання. *(A6/Q5 — найнебезпечніше)*
