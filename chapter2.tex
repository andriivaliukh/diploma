\chapter{Проєктування системи VPN з двофакторною автентифікацією}

У цьому розділі викладено проєктні рішення системи VPN з вбудованою
двофакторною автентифікацією. На основі висновків, сформульованих у
розділі~1, обґрунтовано архітектурний підхід, визначено склад компонентів,
спроєктовано процес автентифікації, модель даних, схему розгортання та
механізми захисту.

\subsection{Вибір монолітної архітектури}

Під час проєктування системи розглядалися два принципово різних
архітектурних підходи: мікросервісна архітектура та монолітна. Мікросервісна
архітектура передбачає декомпозицію системи на незалежні сервіси (автентифікація,
управління сесіями, WireGuard-шлюз), які взаємодіють через мережеві протоколи.
Монолітна архітектура розміщує всі функціональні компоненти в межах одного
серверного процесу.

Для цілей дипломної роботи обрано монолітну архітектуру з таких причин:

\begin{enumerate}
  \item \textbf{Мінімальна складність розгортання}: запуск системи зводиться
    до однієї команди \texttt{docker compose up}; відсутній міжсервісний
    мережевий трафік, що міг би спричинити проблеми з налагодженням.

  \item \textbf{Прозорість для академічної оцінки}: вся логіка зосереджена
    в одній кодовій базі, що дозволяє прослідкувати повний потік виконання
    від HTTP-запиту до операції WireGuard.

  \item \textbf{Відповідність масштабу}: система розрахована на десятки
    користувачів (демонстраційний сценарій), де накладні витрати мікросервісної
    архітектури є невиправданими.

  \item \textbf{Управління станом WireGuard}: інтерфейс \texttt{wg0} є
    локальним ресурсом сервера; розподіл між сервісами ускладнив би доступ
    до нього.
\end{enumerate}

Цей підхід узгоджується з практикою провідних VPN-систем на початкових
стадіях розвитку, а також з принципом YAGNI (\textit{You Aren't Gonna
Need It})~\cite{beck1999extreme}: архітектура відповідає поточним вимогам і
може бути розширена при виникненні реальної потреби.

%% ============================================================
\section{Процес автентифікації}
\label{sec:auth_flow}
%% ============================================================

Процес автентифікації розроблений за принципом обов'язкової двофакторної
перевірки: жодна VPN-сесія не може бути створена без успішного завершення
обох кроків. Це досягається через розмежування токенів з різними правами
доступу.

\subsection{Реєстрація та налаштування TOTP}

Реєстрація користувача включає обов'язкове налаштування TOTP. Після
подання імені та пароля сервер автоматично генерує TOTP-секрет і повертає
його у форматі otpauth:// URI (придатному для імпорту в застосунок-автентифікатор
як вручну, так і через QR-код). Для завершення
реєстрації користувач зобов'язаний підтвердити один код із автентифікатора,
що гарантує коректне налаштування другого фактору.
Для неінтерактивних сценаріїв розгортання \texttt{vpncli} підтримує
прапорець \texttt{--auto-totp} під час реєстрації та \texttt{--totp-secret}
під час входу; реєстрація живого користувача повинна завжди виконуватись
через застосунок-автентифікатор.

До підтвердження TOTP користувач не може увійти в систему~--- прапорець
\texttt{is\_verified} у записі \texttt{totp\_secrets} залишається
\texttt{false}, і спроба входу повертає помилку \texttt{403 Forbidden}.
Тому обхід 2FA на рівні API виключено.

\subsection{Двоетапний вхід}

Вхід в систему реалізовано двома послідовними HTTP-запитами з різними
токенами на кожному кроці:

\begin{enumerate}
  \item \textbf{Перевірка пароля}: клієнт надсилає ім'я та пароль на
    \texttt{POST /api/v1/auth/login}. Сервер перевіряє хеш Argon2id
    і повертає \textit{проміжний токен} (TTL: 5 хвилин, scope:
    \texttt{totp\_verify}). Цей токен дійсний виключно для одного
    ендпоінту~--- верифікації TOTP.

  \item \textbf{Верифікація TOTP}: клієнт надсилає 6-значний код
    на \texttt{POST /api/v1/auth/totp/verify} з проміжним токеном
    у заголовку \texttt{Authorization}. Сервер перевіряє код у вікні
    $\pm1$ часовий крок та повертає \textit{токен повного доступу}
    (TTL: 24 години, scope: \texttt{full}).
\end{enumerate}

Такий дворівневий підхід запобігає підбору TOTP-кодів зловмисником,
що перехопив пароль: без проміжного токена перебір кодів неможливий,
а проміжний токен є короткочасним і обмеженим в правах.

\subsection{Управління токенами та авторизація}

Система використовує дві категорії JWT-токенів~\cite{rfc7519}:

\begin{itemize}
  \item \textbf{Проміжний токен} (\texttt{scope: totp\_verify}):
    генерується після успішної перевірки пароля; TTL — 5 хвилин;
    приймається виключно ендпоінтом \texttt{/auth/totp/verify}.

  \item \textbf{Токен повного доступу} (\texttt{scope: full}):
    генерується після успішної верифікації TOTP; TTL — 24 години
    (налаштовується); надає доступ до всіх захищених ресурсів.
    Для адміністраторів містить додатково \texttt{is\_admin: true}.
\end{itemize}

Токени є stateless~--- сервер не зберігає їх у базі даних, що знижує
потребу в централізованому сховищі сесій (за рахунок відмови від
інвалідації токенів через перелік). Перевірка
здійснюється виключно на основі підпису HMAC-SHA256 та часу
придатності~\cite{rfc7519}.

\section{Інтеграція з WireGuard}
\label{sec:wireguard_integration}
%% ============================================================

\subsection{Модель «сесія як вузол»}

Ключовою архітектурною особливістю системи є безпосереднє управління
WireGuard-вузлами (peers) на основі стану VPN-сесій. На відміну від
підходу DefGuard, де використовується PSK-токен~\cite{defguard2024},
розроблена система застосовує прямолінійніше рішення: вузол WireGuard
з'являється на інтерфейсі \texttt{wg0} виключно після успішної
двофакторної автентифікації і видаляється у момент завершення або
відкликання сесії.

Це забезпечує властивість \textit{session-gated VPN access}:
WireGuard-з'єднання є фізично неможливим без чинної автентифікованої
сесії. Навіть якщо зловмисник отримає клієнтський публічний ключ, без
наявності запису вузла на сервері тунель не буде встановлено.

\subsection{Управління інтерфейсом wg0}

Модуль \texttt{wireguard} керує інтерфейсом \texttt{wg0} через виклики
системних утиліт \texttt{wg} та \texttt{ip} з пакетів \texttt{wireguard-tools}
та \texttt{iproute2}. Відмова від сторонніх Python-бібліотек для WireGuard мінімізує
залежності та забезпечує використання канонічного інструментарію.

Основні операції над інтерфейсом:

\begin{itemize}
  \item \textbf{Додавання вузла}: \texttt{wg set wg0 peer <pubkey>
    allowed-ips <ip>/32}~--- виконується синхронно під час обробки
    запиту \texttt{POST /api/v1/vpn/sessions}.

  \item \textbf{Видалення вузла}: \texttt{wg set wg0 peer <pubkey>
    remove}~--- виконується при відкликанні сесії через API
    або автоматично фоновим завданням при закінченні TTL.

  \item \textbf{Статистика вузла}: \texttt{wg show wg0 dump}~---
    надає поля \texttt{last\_handshake}, \texttt{transfer\_rx},
    \texttt{transfer\_tx} для моніторингу активності клієнта.
\end{itemize}

Прозорість & Потребує розуміння PSK-механізму WireGuard & Зрозуміло: вузол або є, або ні \\
\hline
Стандартний WG-клієнт & Ні (потрібен власний клієнт DefGuard) & Так (будь-який WireGuard-клієнт) \\
\hline
Складність реалізації & Висока (через синхронізацію PSK між сервісами) & Низька (два subprocess-виклики) \\
\hline
\end{tabularx}
\end{table}

\subsection{Розподіл IP-адрес}

Система підтримує підмережу WireGuard (за замовчуванням \texttt{10.10.0.0/24}),
де \texttt{10.10.0.1} є адресою серверного інтерфейсу \texttt{wg0}, а адреси
від \texttt{10.10.0.2} до \texttt{10.10.0.254} формують пул клієнтів
(253 адреси).

Алгоритм розподілу адрес:
\begin{enumerate}
  \item Запит до бази даних для отримання множини IP-адрес всіх
    \textit{активних} VPN-сесій.
  \item Вибір найменшої вільної адреси з пулу.
  \item Атомарне збереження призначеної адреси при створенні запису сесії.
  \item Повернення адреси до пулу при зміні статусу сесії на
    \texttt{expired} або \texttt{revoked}.
\end{enumerate}

\subsection{Фоновий процес очищення сесій}

Модуль \texttt{tasks} реалізує фонове завдання, що запускається кожні
60 секунд під час роботи застосунку. Завдання виконує такі дії:

\begin{enumerate}
  \item Запит до бази даних: вибір сесій зі статусом \texttt{active},
    для яких \texttt{expires\_at < now()}.
  \item Для кожної такої сесії: виклик \texttt{wg set wg0 peer <pubkey>
    remove} та встановлення статусу \texttt{expired} у базі даних.
\end{enumerate}

Цей механізм гарантує, що прострочені сесії не залишаються у
конфігурації WireGuard, навіть якщо клієнт не здійснив явного
відключення.

\section{Модель даних}
\label{sec:data_model}
%% ============================================================

База даних системи містить п'ять сутностей: \texttt{users},
\texttt{totp\_secrets}, \texttt{devices}, \texttt{vpn\_sessions}
та \texttt{system\_settings}. Визначення моделей виконано засобами
SQLAlchemy 2.0 з типізованим ORM-API (Mapped columns).

\subsection{Сутності та їх атрибути}

\textbf{Сутність \texttt{users}} зберігає облікові дані користувача.
Ідентифікатор є UUID~\cite{rfc4122} у рядковому представленні версії UUIDv4,
що практично унеможливлює передбачення наступного ідентифікатора.
Пароль зберігається виключно у вигляді хешу Argon2id~--- оригінальний
пароль ніколи не потрапляє до бази даних. Прапорці \texttt{is\_admin}
та \texttt{is\_active} дозволяють адміністратору керувати доступом
без видалення облікових записів.

\textbf{Сутність \texttt{totp\_secrets}} пов'язана з \texttt{users}
відношенням «один до одного» (UNIQUE-обмеження на \texttt{user\_id}).
TOTP-секрет зберігається зашифрованим алгоритмом Fernet (симетричне
шифрування AES-128-CBC + HMAC-SHA256)~--- відкритий секрет ніколи
не записується до бази даних. Прапорець \texttt{is\_verified}
блокує вхід до підтвердження факту налаштування автентифікатора.

\textbf{Сутність \texttt{devices}} представляє зареєстровані
WireGuard-пристрої користувача. Кожен пристрій ідентифікується
унікальним публічним ключем WireGuard (44 символи Base64 (32 байти ключа
з padding-символом '=')). Унікальне обмеження на пару (\texttt{user\_id},
\texttt{name}) гарантує унікальність імен пристроїв у межах
одного користувача.

\textbf{Сутність \texttt{vpn\_sessions}} зберігає стан кожної
VPN-сесії. Поле \texttt{assigned\_ip} містить IP-адресу, що
виділена клієнту у підмережі WireGuard. Перелічуваний тип
\texttt{status} є частиною бізнес-логіки: значення \texttt{active}
означає наявність відповідного запису у конфігурації WireGuard,
а \texttt{expired} / \texttt{revoked}~--- його відсутність.
Поле \texttt{expires\_at} є абсолютним UTC-часом~--- фонове завдання
порівнює його з поточним часом.

\textbf{Сутність \texttt{system\_settings}} реалізована за патерном
«одиночка» (singleton): обмеження \texttt{CHECK(id = 1)} гарантує
існування рівно одного запису. Параметри \texttt{max\_sessions\_per\_user}
та \texttt{session\_ttl\_hours} впливають на логіку створення сесій
і є змінними через адміністративний API.

\subsection{Діаграма зв'язків між сутностями}

Зв'язки між сутностями:

\begin{itemize}
  \item \texttt{User} $\xrightarrow{1:1}$ \texttt{TOTPSecret}~---
    кожен користувач має рівно один TOTP-секрет;
  \item \texttt{User} $\xrightarrow{1:N}$ \texttt{Device}~---
    користувач може мати кілька зареєстрованих пристроїв;
  \item \texttt{User} $\xrightarrow{1:N}$ \texttt{VPNSession}~---

    публічного ключа).

\subsection{Індекси та обмеження}

Для забезпечення продуктивності запитів та цілісності даних
встановлено такі індекси та обмеження:

\begin{itemize}
  \item Унікальний індекс на \texttt{users.username}~---

  \item Унікальний індекс на \texttt{devices.public\_key}~---

    користувача (використовується при перевірці ліміту сесій);
  \item Унікальне обмеження \texttt{totp\_secrets.user\_id}~---
    гарантія відношення «один до одного».
\end{itemize}

\subsection{SQLite vs PostgreSQL}

Вибір між SQLite і PostgreSQL є ключовим рішенням щодо бази даних.
Аналіз критеріїв для умов дипломного проєкту:

\begin{itemize}
  \item \textbf{Складність розгортання}: SQLite не потребує окремого
    процесу або контейнера~--- база даних є звичайним файлом.
    PostgreSQL вимагає окремого контейнера, конфігурації паролів
    та ініціалізаційних скриптів.

  \item \textbf{Конкурентний запис}: SQLite у WAL-режимі підтримує
    паралельне читання і один запис на момент часу. Для очікуваного
    навантаження (до 10 користувачів, близько 1 запису на хвилину)
    це є цілком достатнім. PostgreSQL надає перевагу при десятках
    одночасних записів.

  \item \textbf{Резервне копіювання}: SQLite~--- копіювання одного
    файлу; PostgreSQL~--- \texttt{pg\_dump} або знімок тому.

  \item \textbf{Переносимість}: SQLite-база є звичайним файлом,
    що спрощує передачу даних між середовищами розробки і
    демонстрації.
\end{itemize}

Враховуючи масштаб та призначення проєкту, SQLite є обґрунтованим
і достатнім вибором. Архітектура через рівень
абстракції SQLAlchemy дозволяє замінити SQLite на PostgreSQL
зміною лише рядка підключення, не модифікуючи бізнес-логіку.

\label{sec:security}
%% ============================================================

\subsection{Захист облікових даних}

Паролі зберігаються виключно у вигляді хешу алгоритму
Argon2id~\cite{biryukov2016argon2}~--- переможця конкурсу Password
Hashing Competition 2015. На відміну від bcrypt, Argon2id є
налаштовуваним за часом і пам'яттю, що забезпечує стійкість до
атак за допомогою GPU та ASIC. Бібліотека \texttt{argon2-cffi} використовує
профіль RFC\_9106\_LOW\_MEMORY за замовчуванням
(час: 3 ітерації, пам'ять: 64 МіБ, паралелізм: 4)~\cite{owasp_password_storage}.

Верифікація пароля здійснюється бібліотекою \texttt{argon2-cffi},
яка реалізує захисне порівняння через функцію з постійним часом
виконання, що запобігає атакам на часові характеристики.

\subsection{Захист TOTP-секретів}

TOTP-секрети є критичними даними~--- їх витік еквівалентний
компрометації другого фактору. Секрети зашифровуються симетричним
алгоритмом Fernet~\cite{cryptography_fernet}: AES-128-CBC для
конфіденційності та HMAC-SHA256 для автентичності. Ключ Fernet
генерується із серверного секрету \texttt{VPN\_SECRET\_KEY}
через функцію виведення ключів HKDF~\cite{rfc5869} і зберігається
виключно у змінній середовища контейнера, не в базі даних.

\subsection{Безпека JWT}

Система використовує два різних токени з явно різними правами:

\begin{enumerate}
  \item Проміжний токен (scope: \texttt{totp\_verify}, TTL: 5 хв)~---
    обмежений одним ендпоінтом; мінімізує вікно можливого зловживання
    при перехопленні.

  \item Токен повного доступу (scope: \texttt{full}, TTL: 24 год)~---
    надається лише після успішного 2FA.
\end{enumerate}

Обидва токени підписані алгоритмом HS256 (HMAC-SHA256) одним і тим самим
секретом \texttt{VPN\_JWT\_SECRET}~\cite{rfc7519}; розрізнення між ними
здійснюється виключно за claim-ом \texttt{scope}, який перевіряється у
відповідному middleware.
Стандарт JWT вимагає, щоб ключ підпису мав ентропію не менше ніж
розмір виводу хеш-функції~--- для HS256 це 256 біт (32 байти)~\cite{rfc7518}.

\subsection{Мережева безпека}

Мережева архітектура системи забезпечує два рівні ізоляції:

\begin{itemize}
  \item \textbf{Зовнішній рівень}: HTTPS (TLS 1.2+) через Caddy
    захищає весь API-трафік між клієнтом і сервером. Порт
    \texttt{8000} (API-сервер) не доступний за межами Docker-мережі~---
    лише Caddy може до нього звертатись.

  \item \textbf{Транспортний рівень}: WireGuard використовує власне
    шифрування на основі ChaCha20-Poly1305~\cite{donenfeld2017wireguard};
    UDP-порт 51820 доступний безпосередньо (без TLS-проксі).
\end{itemize}

\subsection{Валідація вхідних даних}

Всі вхідні дані валідуються через Pydantic-моделі~\cite{pydantic_docs}
до обробки бізнес-логікою:

\begin{itemize}
  \item Ім'я користувача: 3--50 символів, лише алфавітно-цифрові
    та підкреслення.
  \item Пароль: мінімум 8 символів.
  \item Публічний ключ WireGuard: 44 символи, валідний Base64.
  \item TOTP-код: рівно 6 цифр.
\end{itemize}

FastAPI автоматично повертає відповідь \texttt{422 Unprocessable Entity}
для запитів, що не відповідають схемі, не допускаючи їх до
бізнес-логіки. Це виключає широкий клас ін'єкційних атак на рівні
вхідних даних.

\subsection{Контейнери Docker Compose}

Система розгортається засобами Docker Compose і складається
з двох контейнерів:

\textbf{Контейнер \texttt{caddy}} використовує офіційний образ
\texttt{caddy:2} без змін. Caddy виконує роль зворотного проксі з
автоматичним управлінням TLS-сертифікатами через Let's Encrypt~\cite{letsencrypt2015}
для продакшн-середовища або вбудований CA для локальної розробки.
Caddy прослуховує порти 80 (HTTP → HTTPS-перенаправлення) і 443 (HTTPS)
та проксує всі запити до \texttt{vpn-server:8000} через внутрішню
Docker-мережу.

\textbf{Контейнер \texttt{vpn-server}} будується з власного
\texttt{Dockerfile} на основі офіційного образу Python (slim-варіант) з додатковими
пакетами \texttt{wireguard-tools} та \texttt{iproute2}. Контейнер
вимагає двох спеціальних привілеїв:

\begin{itemize}
  \item \textbf{\texttt{NET\_ADMIN}} (\texttt{cap\_add}): можливість
    Linux для управління мережевими інтерфейсами. Без цієї можливості
    виклики \texttt{ip link add} та \texttt{wg set} завершаться
    помилкою \texttt{EPERM}.

  \item \textbf{\texttt{net.ipv4.ip\_forward=1}} (\texttt{sysctls}):
    параметр ядра, що дозволяє пересилання IP-пакетів між інтерфейсами.
    Необхідний для маршрутизації трафіку від WireGuard-клієнтів
    до зовнішньої мережі.
\end{itemize}

API-сервер на порту 8000 \textit{не публікується} (використовується
\texttt{expose}, а не \texttt{ports})~--- доступний лише з Docker-мережі,
куди підключений Caddy. Порт WireGuard 51820/UDP публікується
безпосередньо, оскільки WireGuard-з'єднання не може проксіюватися
через Caddy.

\subsection{Конфігурація через змінні середовища}

Всі чутливі параметри передаються через змінні середовища, що відповідає
принципу 12-Factor App~\cite{heroku_12factor}. Це дозволяє розділяти
конфігурацію для розробки і продакшн-середовищ без зміни коду:

\begin{itemize}
  \item \texttt{VPN\_SECRET\_KEY}~--- ключ для шифрування TOTP-секретів
    (Fernet);
  \item \texttt{VPN\_JWT\_SECRET}~--- ключ для підпису JWT-токенів;
  \item \texttt{VPN\_WG\_ENDPOINT}~--- публічна адреса сервера
    для клієнтів WireGuard;
  \item \texttt{VPN\_WG\_SUBNET}~--- підмережа WireGuard
    (за замовчуванням \texttt{10.10.0.0/24});
  \item \texttt{VPN\_ADMIN\_USERNAME} та \texttt{VPN\_ADMIN\_PASSWORD}~---
    облікові дані початкового адміністратора.
\end{itemize}

\subsection{Постійне зберігання даних}

Система використовує два іменованих Docker-томи для забезпечення
збереження даних між перезапусками контейнерів:

\begin{itemize}
  \item \textbf{\texttt{vpn-data}}: монтується до \texttt{/data}
    всередині контейнера \texttt{vpn-server}; містить файл SQLite-бази
    даних (\texttt{vpnservice.db}) та приватний ключ WireGuard-сервера
    (\texttt{wg\_private.key}).

  \item \textbf{\texttt{caddy-data}}: монтується до \texttt{/data}
    всередині контейнера \texttt{caddy}; містить TLS-сертифікати
    та ключі, керовані Caddy.
\end{itemize}

\subsection{CLI-клієнт як Docker-образ}

Клієнтський інструмент \texttt{vpncli} розповсюджується як окремий
Docker-образ (\texttt{Dockerfile.cli}) на основі офіційного образу Python (slim-варіант)
з вбудованим \texttt{wireguard-tools}. Контейнер CLI також вимагає
можливості \texttt{NET\_ADMIN} для управління WireGuard-інтерфейсом
всередині контейнера:

\begin{lstlisting}[language=bash, caption={Запуск CLI-клієнта у Docker-контейнері},
                   label={lst:cli_docker}]
docker run --rm -it --cap-add NET_ADMIN \
  vpncli connect --server https://vpn.example.com
\end{lstlisting}

Такий підхід усуває залежності на стороні клієнта: для використання
VPN-клієнта достатньо наявності Docker на машині кінцевого користувача.

%% ============================================================
\section{Висновки до розділу~2}
\label{sec:chapter2_conclusions}
%% ============================================================

У цьому розділі розроблено повну архітектуру системи VPN
з двофакторною автентифікацією на основі WireGuard.

Передусім, обрана монолітна архітектура на основі FastAPI є оптимальною
для масштабу та цілей проєкту: єдина кодова база, запуск однією командою,
прозорий потік виконання без міжсервісних залежностей.

Окрім цього, двоетапний процес автентифікації з токенами різного рівня
доступу (проміжний \texttt{totp\_verify} та повний \texttt{full})
архітектурно унеможливлює обхід двофакторної перевірки. WireGuard-вузол
з'являється на сервері виключно після успішного проходження обох кроків.

Водночас, модель «сесія як вузол» є простішою та прозорішою за підхід
DefGuard з PSK-токенами, забезпечуючи аналогічний рівень безпеки
при нижчій складності реалізації та підтримці стандартного WireGuard-клієнта.

Додатково, вибір SQLite у WAL-режимі, Argon2id для паролів, Fernet для
TOTP-секретів та JWT з розмежуванням scope формує зрілий стек безпеки,
що відповідає актуальним рекомендаціям OWASP і використовує стандартизовані
алгоритми~\cite{owasp_password_storage, rfc7519, rfc6238}.

Зрештою, архітектура розгортання через Docker Compose з двома контейнерами
забезпечує чіткий поділ відповідальності між TLS-проксі (Caddy) і
серверним застосунком, відповідає принципу мінімальних привілеїв
(NET\_ADMIN тільки там, де потрібно) та дозволяє запустити повну систему
в одну команду на будь-якому хості з підтримкою Docker.
